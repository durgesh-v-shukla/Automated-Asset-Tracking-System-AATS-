ISSUE #2: ASYNC IN THREAD DEADLOCK (STABILITY CRITICAL)
─────────────────────────────────────────────────────────
Location: student_agent/bluetooth_monitor.py, line 62
Severity: CRITICAL
Risk: Calling asyncio.run() inside a daemon thread will crash if:
       - Thread is launched during agent startup
       - Main event loop already exists in the process
       - Result: "asyncio.run() cannot be called from a running event loop" exception

Current Code:
    def _monitor_loop(self) -> None:
        if BleakScanner is None:
            print("Bleak is not installed. Bluetooth monitor disabled.")
            return
        while self.running:
            try:
                seen = asyncio.run(self._scan_devices())  # ← CRASH HERE
            except Exception as exc:
                print(f"Bluetooth scan error: {exc}")
                seen = {}

Impact: Bluetooth device scanning will crash periodically, breaking asset monitoring
        for Bluetooth-tracked devices. Silent failure in exception handler.

Remediation Plan:
STEP 1: Use asyncio.new_event_loop() for thread isolation
    - Replace asyncio.run() with proper event loop creation
    - Create loop in the thread context where it's needed
    
Fixed Code:
    def _monitor_loop(self) -> None:
        if BleakScanner is None:
            print("Bleak is not installed. Bluetooth monitor disabled.")
            return
        
        loop = asyncio.new_event_loop()
        asyncio.set_event_loop(loop)
        
        while self.running:
            try:
                seen = loop.run_until_complete(self._scan_devices())
            except Exception as exc:
                print(f"Bluetooth scan error: {exc}")
                seen = {}
            finally:
                time.sleep(self.poll_interval_sec)
        
        loop.close()

STEP 2: Test with both USB and Bluetooth devices configured
    - Verify Bluetooth scanner works without crashing
    - Monitor for "asyncio already running" errors in logs

Estimated Effort: 30 minutes
Dependencies: None (Phase 1, can start immediately)


ISSUE #3: NO RATE LIMITING ON LOGIN (DOS VULNERABILITY)
─────────────────────────────────────────────────────────
Location: server/app.py, POST /auth/login endpoint
Severity: CRITICAL
Risk: Attacker can brute-force admin password with no throttling. No delay between
       login attempts, no account lockout, no IP-based rate limiting.

Current Implementation:
    - Simple username/password comparison
    - No rate limiter on endpoint
    - No logging of failed attempts

Impact: Attacker can try 1000s of passwords per second without consequence.
        Password security reduced to single-digit entropy.

Remediation Plan:
STEP 1: Install slowapi dependency
    pip install slowapi
    Add to server/requirements.txt

STEP 2: Configure rate limiter in server/app.py
    from slowapi import Limiter
    from slowapi.util import get_remote_address
    
    limiter = Limiter(key_func=get_remote_address)
    app.state.limiter = limiter

STEP 3: Apply rate limiter to /auth/login
    @app.post("/auth/login")
    @limiter.limit("5/minute")  # 5 attempts per minute per IP
    async def login(request: Request, credentials: LoginRequest) -> LoginResult:
        # ... existing code ...

STEP 4: Log failed attempts
    - Add logger.warning() for failed login attempts
    - Include timestamp, username, IP address
    - Use for intrusion detection/alerts

Estimated Effort: 45 minutes
Dependencies: Phase 1 - None (can start immediately)


ISSUE #4: NULL event_id WITH UNIQUE CONSTRAINT (DATA INTEGRITY)
────────────────────────────────────────────────────────────────
Location: server/database.py, line 25; student_agent device monitors
Severity: CRITICAL
Risk: event_id column is marked UNIQUE but can be NULL. Multiple NULL values 
       will cause "UNIQUE constraint failed" silently (INSERT OR IGNORE behavior).
       Events silently fail to insert, breaking audit trail.

Current Schema (database.py line 25):
    CREATE TABLE IF NOT EXISTS device_events (
        id INTEGER PRIMARY KEY AUTOINCREMENT,
        event_id TEXT UNIQUE,  # ← Can be NULL, violates UNIQUE semantics
        ...
    )

Current Code (device_monitor.py, bluetooth_monitor.py):
    - Do NOT generate event_id
    - Pass None (or omit) event_id field
    - Server receives event_id=None
    - Database insert fails silently due to INSERT OR IGNORE

Impact: Event history is incomplete. Critical alerts may not be recorded.
        Audit trail is unreliable for incident investigation.

Remediation Plan:
STEP 1: Generate UUID in student agent (preferred approach)
    Add to both device_monitor.py and bluetooth_monitor.py:
    
    import uuid
    
    def _emit(self, payload: Dict) -> None:
        if "event_id" not in payload:
            payload["event_id"] = str(uuid.uuid4())
        if self.on_change_callback:
            self.on_change_callback(payload)

STEP 2: Add NOT NULL + DEFAULT constraint to event_id
    Modify server/database.py _init_db() method to ensure event_id is required

STEP 3: Test event insertion
    - Publish 100 events and verify all are recorded
    - Check database for NULL event_id entries
    - Verify no "UNIQUE constraint failed" errors in logs

Estimated Effort: 1 hour
Dependencies: Phase 1 - can start immediately

ISSUE #6: GLOBAL STATE RACE CONDITION (THREADING)
──────────────────────────────────────────────────
Location: server/app.py, lines 140-145 (pending_watcher function)
Severity: HIGH
Risk: pending dict is modified without full lock coverage. Race condition 
       between _pending_lock usage and dict modification.

Current Problematic Code:
    with _pending_lock:
        for key, item in pending.items():
            if item["confirmed"]:
                continue
            if now - item["started_at"] >= item["timeout_sec"]:
                item["confirmed"] = True
                to_confirm.append((key, item))
    
    # Lock RELEASED - another thread could modify pending[key] now
    for _, item in to_confirm:
        payload = item["payload"]  # ← Could be stale or deleted

Impact: Double-promotion of alerts or missing alert promotion.
        Unreliable pending window behavior.

Remediation Plan:
STEP 1: Reduce lock-free window
    Do NOT release lock until to_confirm list is finalized
    Keep lock held during entire confirmation process.

Estimated Effort: 45 minutes
Dependencies: Phase 2
