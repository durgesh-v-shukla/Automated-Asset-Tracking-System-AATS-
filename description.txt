PROJECT OVERVIEW
=================
AATS (Automated Asset Tracking System) is a lab anti-theft monitoring system built around three cooperating layers:
- a student agent that runs on each lab PC and watches USB and Bluetooth assets,
- a FastAPI server that receives MQTT data, applies debounce logic, stores history in SQLite, and exposes admin APIs,
- and a browser-based admin dashboard that shows live status, alerts, and event history.

The project is designed for a real demo flow:
1. a lab PC runs the agent,
2. the agent publishes heartbeats and device state changes to MQTT,
3. the server turns temporary disconnects into WARNING/PENDING states first,
4. and if the problem persists, the state becomes CRITICAL on the dashboard and a database record is stored.

CODE STRUCTURE
==============
The code is split into setup scripts, runtime code, and presentation code.

student_agent/
- main.py is only the launcher.
- service_runner.py contains the actual agent runtime.
- mqtt_client.py publishes retained PC status and non-retained device events.
- device_monitor.py watches configured USB devices.
- bluetooth_monitor.py watches configured Bluetooth devices.
- windows_service.py exposes the agent as a Windows service called AATSAgentService.

server/
- app.py defines the FastAPI app, the authentication route, the public read APIs, the delete/removal APIs, and the debounce workflow.
- config.py reads environment variables such as the MQTT broker, database path, admin credentials, and timeout values.
- database.py creates the SQLite tables and implements all insert/update/query methods.
- mqtt_listener.py connects to MQTT, subscribes to the status and event topics, and passes decoded JSON into app.py.
- inspect_db.py is a helper for printing event history, current device state, and PC heartbeat summaries for screenshots and debugging.
- models.py defines the Pydantic response models used by the API.

admin_dashboard/
- login.html is the login screen.
- index.html is the lab picker.
- dashboard.html is the per-lab monitoring page.
- script.js fetches data from the server, renders the UI, stores the admin token, refreshes the page, and plays the critical alert sound.
- styles.css defines the dashboard layout, colors, cards, badges, tables, and responsive presentation.

SETUP SCRIPTS
=============
admin_setup.py prepares the admin machine.
It can:
- find an installed Mosquitto broker,
- fall back to a bundled broker binary when present,
- optionally install Mosquitto through winget or a verified download path,
- open firewall ports for MQTT, the API, and the dashboard,
- start the broker and the FastAPI server,
- serve the dashboard,
- open the browser automatically,
- and broadcast the admin PC IP so lab agents can discover it during first run.

agent_setup.py prepares each lab PC.
It can:
- listen for the admin broadcast for up to 30 seconds,
- fall back to manual IP entry if discovery fails,
- prompt for a unique PC ID,
- scan connected USB devices so the user can choose what to track,
- write student_agent/config.json,
- and register the agent as either a Windows service or a registry startup entry.

RUNTIME BEHAVIOR
================
The runtime flow is straightforward but important:
- student_agent/service_runner.py loads the JSON config and starts the MQTT client.
- It launches the USB monitor and Bluetooth monitor in background threads.
- It publishes a periodic online heartbeat every configured interval.
- When a tracked device changes state, the callback packages the event with lab_id, pc_id, device_id, device_type, status, timestamps, and optional RSSI.
- The event is published to MQTT.
- The server's MQTT listener receives the message and calls the proper handler.
- The server decides whether the state is immediately safe, should wait in PENDING, or should become CRITICAL after a timeout.
- The dashboard refreshes from the API and renders the latest state.

DETAILED CODE BEHAVIOR
======================
Agent side:
- USB monitoring is based on Windows Get-PnpDevice output and compares InstanceId values against configured VID/PID pairs.
- Bluetooth monitoring uses bleak to scan nearby devices and compares RSSI values against a configured weak-signal threshold.
- MQTT status messages are retained so the latest PC heartbeat is always available.
- MQTT event messages are not retained because they represent history rather than state.
- The agent publishes an offline status on shutdown, while the MQTT Last Will also handles unexpected loss of connection.
- When run as a service, the runtime is stopped through the Windows service wrapper.

Server side:
- `POST /auth/login` checks the configured admin username and password and returns a token derived from the admin password.
- `GET /health` returns a simple status/time response for liveness.
- `GET /labs` lists labs and summary counts.
- `GET /labs/{lab_id}/pcs` returns heartbeat data and also marks stale PCs as offline using the configured heartbeat timeout.
- `GET /labs/{lab_id}/devices` returns the current live device state.
- `GET /alerts` and `GET /events` return historical records with filters.
- `DELETE /labs/{lab_id}/pcs/{pc_id}` removes a PC from tracking.
- `DELETE /labs/{lab_id}/pcs/{pc_id}/devices/{device_id}` removes one device from tracking.
- The server keeps an in-memory pending map for debounce scheduling and also stores pending windows in SQLite so they survive restart.
- When a pending window expires, the device is promoted to CRITICAL and a details_json payload records the debounce duration.

Database layer:
- device_events stores the forensic timeline.
- device_state_current stores the latest dashboard snapshot.
- pc_heartbeat stores online/offline state and agent version.
- pending_window stores active debounce windows.
- excluded_pcs and excluded_devices track items that the user removed from monitoring.

Dashboard behavior:
- login.html authenticates the user and stores a token in localStorage.
- index.html queries the server for labs and shows them as selectable cards.
- dashboard.html loads one lab's PCs and devices.
- script.js attaches the x-admin-token header to requests, renders the card UI, refreshes periodically, and plays a sound the first time a new critical event appears.
- styles.css uses a clean card-based interface with severity colors and responsive layout.

DATA AND API SHAPE
==================
Important MQTT topics:
- aats/lab/{lab_id}/pc/{pc_id}/status
- aats/lab/{lab_id}/pc/{pc_id}/event

Important HTTP routes:
- /health
- /auth/login
- /labs
- /labs/{lab_id}/devices
- /labs/{lab_id}/pcs
- /labs/{lab_id}/pcs/{pc_id}
- /labs/{lab_id}/pcs/{pc_id}/devices/{device_id}
- /alerts
- /events

Useful table columns:
- device_events: event_id, lab_id, pc_id, device_id, device_label, device_type, status, severity, alert_status, observed_at, agent_time, received_at, source, details_json
- device_state_current: lab_id, pc_id, device_id, device_label, device_type, current_status, severity, alert_status, rssi, pending_since, updated_at
- pc_heartbeat: lab_id, pc_id, pc_status, last_seen, agent_version, updated_at
- pending_window: lab_id, pc_id, device_id, started_at, timeout_sec, confirmed, payload_json

WHAT THE PROJECT DEMONSTRATES
=============================
AATS demonstrates an end-to-end IoT monitoring chain:
- sensing at the edge,
- messaging over MQTT,
- server-side alert logic,
- persistent storage,
- and a dashboard that can be used for live monitoring and report screenshots.

The demo scenarios are the same ones used in the README:
- USB removal becomes warning first, then critical if the device stays disconnected.
- Bluetooth moving out of range becomes weak signal first, then critical if it stays away.
- An abrupt PC shutdown appears as offline through MQTT Last Will and Testament.

LIMITATIONS
===========
The current code is still a lab prototype:
- it uses a single shared admin account,
- MQTT and HTTP are not TLS-protected by default,
- and SQLite is good for a small deployment but not for a large campus-scale system.

steps. This project is a lab prototype and is NOT production-ready until the
Phase 1 critical fixes are implemented and validated.
